# 未分组 → 真文件夹 + 多选批量操作 — 设计

日期:2026-06-22
范围:本期只做 ①②。③「一个位置被多条链路收藏」留作下一期单独 spec(不在本设计内)。

## 背景与问题

新打的标签默认散落在 `store.tree` 根级(`addTag` 不带 parentId)。侧边栏显示的「(未分组)」**不是真实文件夹节点**,而是 `adapter.ts` 把根级散标签临时合成出来的一个组(`LOOSE_TOUR_ID = "__loose__"`)。因为它没有真实 id,所以:

- 不能重命名(`renameFolder` 拦截 `LOOSE_TOUR_ID`)
- 不能拖进别的文件夹(`handleDrag` 排除 `__loose__`)
- 不能删除(`deleteFolder` 拦截)
- 不能在其下新建子文件夹(`createSubFolder` 拦截)

结果:这个桶里的标签没法当成一个文件夹来批量管理。

同时,多选批量「复制 / 删除」缺失:右键删除/复制命令只对焦点那一个节点生效,没有消费 VS Code 多选传入的选中数组。

## 目标

① 把「未分组」变成一个**真实文件夹**,可改名 / 可拖进别的文件夹 / 可删 / 可建子文件夹,同时仍然承担「新标签落脚点」的角色。
② 多选**拖动 / 复制 / 删除**批量生效。

非目标(本期不做):一个标签复制多份放进不同文件夹(③ 多链路模型)。

## 设计 ①:未分组 → 真文件夹(带「收件箱」身份)

### 数据模型

给 `FolderNode` 增加可选字段:

```ts
inbox?: boolean;   // 该文件夹是「新标签收件箱」
```

从此**所有标签都活在文件夹里**,根级不再允许散标签。合成组 `__loose__`(`LOOSE_TOUR_ID` / `LOOSE_TITLE`)彻底移除。

### 收件箱规则(单一真源)

新增 `getOrCreateInbox(store, idGen): FolderNode`:

- 返回根级那个 `inbox === true` 的文件夹;
- 若不存在,**懒创建**一个:`{ type:"folder", inbox:true, title:"未分组", children:[] }`,插在**根级顶部**(index 0,沿用旧「(未分组)」置顶的位置),返回它;
- 始终最多一个收件箱(创建前先找,找到就复用)。

新标签流程(`recorder/commands.ts` 原 `addTag(getStore(), tag)`)改为:取 `getOrCreateInbox(...).id` 作为 parentId 落进收件箱。

### 毕业(graduation)

收件箱在以下任一操作后**清掉 `inbox` 标记**,降为普通文件夹(它已归档的标签留在原地不动):

1. **改名**:`renameFolder` 成功改名后,若该文件夹 `inbox`,清掉。
2. **被拖出根级**:`handleDrop` / `moveNode` 把它挪到「parent ≠ 根」时,清掉。(在根级内重排序不算毕业。)
3. **删除**:随文件夹一起进回收站,`inbox` 标记随之消失。

毕业后,下次打新标签时 `getOrCreateInbox` 会在根级懒创建一个新的空「未分组」。即「改名即毕业,旁边另长新篮子」。

### 迁移(打开旧 store 时一次性、幂等)

`parse` / 加载后执行 `migrateLooseTags(store, idGen)`:

- 若根级存在任何 `type==="tag"` 的散标签 → 新建一个 `inbox:true` 的「未分组」文件夹(置顶),把这些散标签按原顺序搬进去;
- 若根级无散标签 → 不动(不强行造一个空收件箱;收件箱按需懒创建)。
- 幂等:执行后根级再无散标签,重复执行无副作用。

### 拖放调整(`player/tree/index.ts`)

- 删除 `handleDrag` / `handleDrop` / `subfolderNodesOf` / `getParent` 里所有 `"__loose__"` 特判。
- **标签拖到空白处**(target 为 undefined):原语义=变根级散标签 → 改为**落进收件箱**(`getOrCreateInbox`)。
- **文件夹拖到空白处**:仍留在根级(folders 可以住根级)。
- 拖动一个 `inbox` 文件夹到某个文件夹下(parent ≠ 根)→ 触发毕业(清 `inbox`)。

### 解除三处拦截(`lodestar/commands.ts`)

`renameFolder` / `deleteFolder` / 新建子文件夹 里针对 `LOOSE_TOUR_ID` 的 `showInformationMessage` 拦截全部移除。收件箱与普通文件夹一视同仁。

## 设计 ②:多选 批量 拖动 / 复制 / 删除

`treeView` 已设 `canSelectMany: true`。

- **拖动**:已可用(`handleDrag(source: TreeItem[])` 收集全部 id → `handleDrop` 逐个 `moveNode`)。① 落地后回归验证一遍。
- **删除**:新增 `deleteSelection(focused, selected[])`。VS Code 多选时命令第二参数为选中节点数组;命令收集其中的标签 + 文件夹,一并 `removeToTrash`,弹**一个汇总确认框**(例:「删除 3 个标签、1 个文件夹(及其中的标签)?」)。去重:若选中项里既有某文件夹又有其子节点,只删文件夹即可(子节点随之进回收站)。
- **复制**:新增 `copySelection(focused, selected[])`。收集选中项(文件夹递归展开成其全部标签)的 markdown 链接,一次性写剪贴板,状态栏提示条数。
- `package.json` 右键菜单:删除 / 复制指向新的批量命令(单选时数组长度为 1,行为与原先一致)。

## 测试(TDD)

纯逻辑先写 vitest 测试再实现:

- `getOrCreateInbox`:无收件箱时创建并置顶 + flag;已有时复用;不重复创建。
- `migrateLooseTags`:根级散标签搬入收件箱、保序、置顶;无散标签时不动;幂等。
- 毕业判定:改名 / 移出根级 后 `inbox` 被清;根级内重排不清。
- `treeToTours` / `treeToAllTours`:不再产出 `__loose__` 组;所有标签经由真实文件夹覆盖。
- 批量收集:`deleteSelection` / `copySelection` 的「选中集合 → 标签集合」收集与去重(纯函数抽出)。

VS Code 胶水层(命令注册、drop 路由、确认框)薄,逻辑靠上述纯函数覆盖。现有 44 个测试须全绿 + 新增。

## 边界与已知取舍

- 收件箱**按需存在**:空收件箱被毕业/删除后不立即重建,下次打标签才出现。符合「另长新篮子」语义。
- 同一时刻最多一个收件箱;迁移与 `getOrCreateInbox` 都保证唯一。
- ③ 多链路(同一位置进多个文件夹 / 各自命名)不在本期;数据模型保持严格树状,为下一期留空间。
