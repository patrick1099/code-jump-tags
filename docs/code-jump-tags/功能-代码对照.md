# Code Jump Tags 功能 ↔ 代码对照（给 AI 自己看的索引）

> 目的：用户用自然语言指一个「功能/组件」时，我（未来的 Claude）能立刻定位到对应代码，
> 不用每次从头摸。用户说的功能名往往很口语（比如「悬浮小字」「+号」「回收站展开」），
> 下面的「用户怎么说」列就是给我做模糊匹配用的。
>
> 本仓库是 microsoft/codetour 的 fork，被改造成「标签优先的代码跳转工具」。
> 维护本文件：**改了某功能就顺手更新这里的行**，保持可信。

> **怎么用本文件**：用户让我读完它之后，他在新对话里只要甩一个口语名词（「注释框」「文件夹」
> 「小字」「+号」「回收站」…），我应当能立刻答出那是什么、代码在哪、相关坑是什么。
> 下面 §0.5 术语速查就是为这个场景准备的——先扫它，再按需深入对应章节。

## 0. 一句话心智模型（先读这个）

- **唯一真源 = `.code-jump-tags/store.json`**（工作区内）。结构是一棵 `tree`：文件夹(folder) + 标签(tag)，外加一个 `trash` 回收站数组。
- CodeTour 的 `store.tours`（mobx 可观察）被降级成**单向派生缓存**：每次存盘后由 `treeToTours()` 重算。UI 的装饰/CodeLens/树读的是这个派生缓存，**不要反向写它**。
- 概念映射：CodeTour 的 **tour ↔ Code Jump Tags 文件夹**，**step ↔ 标签**。散标签(根下不在任何文件夹)被聚合成一个合成 tour `(未分组)`，id 后缀 `__loose__`。
- tour.id 格式：`${workspaceId}::${folderId | __loose__}`。要从树节点拿真实 folderId，就 `tour.id.split("::")[1]`。
- **纯逻辑层 `src/lodestar/*`（除 persistence/commands/editThread 外）不 import vscode**，所以能用 vitest 跑单测（`test/lodestar/*.test.ts`）。

**关键不变量**：`saveStore()` = 写文件 + `rebuildTours()`。任何改了 store 的命令最后都要 `await saveStore()`，否则 UI 不刷新。

**文件夹嵌套（0.2.0 起）**：数据层 `TreeNode` 本就递归，`findNode`/`moveNode`/`removeNode` 也递归，所以
`renameFolder`/`deleteFolder`/`copyFolderLinks`/拖拽/上下移动标签在嵌套下**自动生效**（都按 `findNode(folderId)`
定位）。渲染：`store.tours` 派生缓存只到顶层文件夹（它喂树根 + tour 选择器，子文件夹不能当顶层可播放
tour），子文件夹靠树视图 `CodeTourTreeProvider.getChildren` 在展开某文件夹时用 `findNode(folderId)` 回原始树取
子文件夹，逐个 `folderToTour()`（已导出）包成 `CodeTourNode` 递归渲染；该文件夹的直属标签仍走
`element.tour.steps`。子文件夹**只是分组**，不是可播放 tour。建文件夹用 `newSubfolder` → `createFolder(..., parentId)`。

**装饰必须看 `allTours` 而非 `tours`（0.3.0 修）**：`store.tours` 只含顶层文件夹，若装饰层读它，嵌套子文件夹里
的标签就**没有行内标记/CodeLens/hover**。所以新增 `treeToAllTours()`（递归收集所有深度的文件夹标签 + 根松散
标签）→ 存进 `store.allTours`（`rebuildTours` 里和 `tours` 一起重算），`decorator.ts` 的 `getTourSteps` 与 reaction
都改读 `store.allTours`。`tours`/`allTours` 分工：前者=树根/选择器（仅顶层），后者=装饰源（全量）。

**任意拖动文件夹（0.3.0）**：`handleDrag` 现在也把 `CodeTourNode`（文件夹，排除 `__loose__`）的 folderId 放进
拖拽数据；`handleDrop` 落到文件夹=进该文件夹，落到标签=成其兄弟，落到空白=回根。`moveNode` 对文件夹同样有效；
**守卫** `isSelfOrDescendant(store, draggedId, targetParentId)` 拒绝把文件夹拖进自己/自己的子孙（否则会把子树丢掉）。

**标签随上方编辑跟着代码走（0.3.3 修）**：gutter 图标用 `setDecorations` 设的 Range，VS Code 会**随文档编辑自动平移**；
但上方小字是 `TagCodeLensProvider` 每次从**存盘行号** `step.line` 重新算的 CodeLens，**不会自动平移** → 在标签上方插入/
删除行时两者脱节（图标走了、小字钉在原行）。修法：`decorator.ts` 监听 `onDidChangeTextDocument`，把每个改动折算成
`LineEdit{start,end,delta}`，对该文档里每个标签调 `shiftedLine(line0, edits)`（纯函数，在 `tree.ts`，规则=改动**结束行严格
在标签行之上**才平移，镜像 gutter 的 ClosedClosed 行为）更新 `TagNode.line`，再 `rebuildTours()` 立即重绘（图标+小字都从同一
更新后的行号画，自然同步）。落盘用 `debounce(800, saveStore)` 不在每次按键写盘。**顺带修了持久化 bug**：以前行号没更新，
reload 后图标会跳回旧行。
锚点取**标签行行首 `(L,0)`**：平移条件 = 改动 `end < L` **或** `end === L 且 endChar === 0`（0.3.4 补）。少了 `endChar`
那一项，**行首换行**（在第 0 列敲回车，把本行代码顶下去）会被漏掉——改动结束在本行、列为 0，标签不动而代码下移。

**gutter 装饰必须每次重画、不能靠自动跟踪（0.3.5 修）**：只在「标签行号变了」时 `rebuildTours()` 重画是不够的——当你
**在标签行内部敲换行**(行中/行尾回车)，标签行号不变(`changed==0`)却没重画，VS Code 会把那条 `Range(L,0,L,1000)` 装饰
**跨行撑大**(行内插换行把区间尾推到下一行)，连敲几个回车 gutter 图标就**糊成一片多行**。修法:`trackLineShifts` 里只要
本次编辑有增减行(`delta≠0`)，即便没有标签移动，也强制 `updateDecorations(activeEditor)` 把 gutter 按存盘行号重画成干净单行。

**整体覆盖文件后标记不漂（0.3.6 修）**：把一个文件整体复制出去改完再整体粘回来时，VS Code 报的是**一整块** contentChange，
`shiftedLine` 的增量平移**算不出**标签的新行（它不知道行内容搬到哪了），于是 `step.line` 存的是旧行。问题在于**显示**和**跳转**走两套定位：
跳转 `gotoLocation` 早就用 `resolveLine(text, line, pattern)` 按**行内容**找回真行(`pattern`=建标签时存的「行首空白 + 转义后的 trim 行文本」正则)，
而显示(gutter 准星 + 小字)一直直接读 `step.line` → 点击跳对、准星钉旧行。修法把两套对齐：
①`decorator.ts` `getTourSteps` 算显示行改成 `resolveLine(contents, step.line, step.pattern)-1`，显示和跳转用**同一套内容找回**；
②`trackLineShifts` 每个标签改调 `reanchorTag(text, {line,pattern}, edits)`（`lodestar/relocate.ts`，纯函数）=先 `shiftedLine` 增量平移、
再让 `resolveLine` 用内容覆盖错误猜测、最后用**当前行文本**刷新 `pattern`，**line 和 pattern 都落盘**，锚点不会越用越旧。
建标签存 `pattern` 的公式抽成 `linePattern(lineText)`（同文件，建标签 `recorder/commands.ts` 与刷新锚点共用，单一真源）。
单测 `test/lodestar/relocate.test.ts` 覆盖 `linePattern`/`reanchorTag`/`resolveLine`（含整体覆盖→内容找回那条）。
**死角（0.3.6 时的上限，0.5.0 已大幅缓解）**：标签行**自身文本小改**（插一个字符、删一个词间空格）在 0.3.6 会因正则**逐字精确**匹配当场失配；且失配后 `resolveLine` 取**全文件第一个**命中，文件里有重复代码（被 `#if 0` 关掉的孪生函数）时标签会被**远处副本**吸走。0.5.0 修这两类。

**就近 + 模糊重定位（0.5.0）**：在 0.3.6 「内容找回」之上加两件事——**容忍小改**和**优先就近**。核心仍是纯函数 `src/lodestar/relocate.ts`，全程不 import vscode：
- **相似度**：`normalizeWs`（trim + 连续空白压一个空格）+ `similarity`（空白归一化后的 Levenshtein 归一化相似度，两串各截断 `MAX_CMP_LEN=200`，两空串记 1）。阈值 `SIMILARITY_THRESHOLD=0.9`。
- **距离优先同心扩圈** `resolveLineFuzzy(text, centerLine, anchorText)`：①空锚点→原行；②中心行相似度 ≥0.9→**信中心、不搜**（正常编辑恒走这条，模糊根本不触发）；③否则按 `SEARCH_RADII=[8,40,Infinity]` 逐圈在 `[center±R]` 找 ≥0.9 候选，**第一圈非空即返回**（圈内取相似度最高、平手取最近）；④全文件无→原行不动。「近处合格的先停」让被改过的近行（模糊~0.9）胜过远处精确副本（宏孪生），消除乱跳。
- **加性数据模型**：`TagNode`/`CodeTourStep` 在原 `pattern`（正则，留给 URL 深链 + 继承自 CodeTour 的 step 层，**不动**）旁**新增 `text`**（原始 trim 行文字，喂模糊解析器）。`lineAnchorText(lineText)` 生成 `text`，与 `linePattern` 并存、建标签/刷新锚点时同步更新（`recorder/commands.ts`、`reanchorTag`）。
- **旧数据回填**：`backfillAnchorText(store)` 在 `persistence.loadStore` 里（`migrateLooseTags` 之后）对只有 `pattern` 没 `text` 的旧标签由 `patternToText`（剥 `^[^\S\n]*` 前缀、反转义）反推出 `text`，`pattern` 原样保留，幂等。
- **统一接入** `resolveAnchoredLine(text, line, anchorText?, pattern?)`：`text` 优先走模糊、否则旧 `pattern` 走正则、再否则信行号。消费方：`decorator.ts` 显示行、`editThread.ts`、`reanchorTag`。`decorator.ts` 的 `else if (step.pattern)` 正则兜底与 URL `gotoLocation` 仍用 `pattern`，不变。
- **两层分工**：实时增量追踪（`shiftedLine` + RC3）是**主力、字符级精确**，处理所有「我们亲眼看见的编辑」；模糊只是**恢复兜底**，仅在「没看见的变动」（关窗时被改、git pull/checkout、外部工具、重开首帧）时才发力。
- **RC3 行内编辑刷新**（`decorator.ts` `trackLineShifts`）：去掉 `delta===0` 早退，改为只把**被改那几行**上的标签锚点（`text`+`pattern`）刷成当前行文字（脏检查后才 `rebuildTours`+存盘，否则只重绘）。锚点永不过期，杜绝逐字改写后下一次结构编辑跳走。
- **拆行**（一行从中间敲回车拆两半）：无需额外代码——换行点在行中（endChar≠0）`shiftedLine` 不移行，模糊「全文件无 ≥0.9 → 不动」保证留在**上半行**并重锚到上半行。
- **死角仍在**：标签行文本被**大改**（相似度跌破 0.9 且全文件无近似）仍恢复不了——内容锚点固有上限，只是阈值化后门槛高了。
- 单测 `test/lodestar/relocate.test.ts` 覆盖 `similarity`/`normalizeWs`/`resolveLineFuzzy`（含「中心 stale 时近模糊胜远精确副本」真正走扩圈的用例）/`lineAnchorText`/`patternToText`/`backfillAnchorText`/`reanchorTag`(text 优先 + 回退旧正则 + 拆行留上半)/`resolveAnchoredLine`。RC3 与拆行的 GUI 行为靠构建 + 手动重载验。

## 1.5 未分组真文件夹 + 多选批量操作（0.4.0 新增）

**「未分组」从合成数据变成真文件夹（0.4.0）**：以前「未分组」是 adapter 动态合成的虚 tour（`__loose__` id 后缀），数据层没有对应文件夹节点。0.4.0 把它升格成真 `FolderNode`（设 `inbox?: boolean` 标记）：

- **新标签落收件箱**（`src/recorder/commands.ts` 的 `addTag` → `createFolder(..., parentId=null)` 时若无「未分组」则自动创建，无 parentId=根级）。
- **旧数据自动迁移**：`src/lodestar/persistence.ts` 的 `loadStore` 在加载时调 `migrateLooseTags(store)` 把散落的根级标签归入新建的「未分组」文件夹。
- **毕业规则**（`src/lodestar/tree.ts` 的 `getOrCreateInbox` / `renameFolderNode` / `moveNode` / `migrateLooseTags`）：一旦用户改名或拖出根级，就清除 `inbox` 标记变成普通文件夹；下次新建标签时若根级没有 inbox 标记的文件夹，自动新建一个空的「未分组」继续用。
- **adapter 移除 `__loose__`**（`src/lodestar/adapter.ts`）：不再合成虚 tour，改从真树里取 inbox 文件夹（若存在）。

**侧边栏多选批量操作（0.4.0）**：
- **选择机制**：tree provider（`src/player/tree/index.ts`）支持 `setSelection()` 多选多个标签/文件夹节点。
- **批量纯函数**（`src/lodestar/selection.ts`）：
  - `pruneCovered(selected, store)` —— 若选中了文件夹 F 和 F 下的标签/子文件夹，只返回 F（去重）。
  - `collectTagsUnder(nodeIds, store)` —— 递归收集所有选中节点（及子树）下的所有标签。
- **三大命令消费多选**：
  - `deleteTag` / `deleteFolder`（`src/lodestar/commands.ts`）：先 `pruneCovered` 再 `confirmDelete` 弹汇总框（「删除 N 个标签 + M 个文件夹？」），确认后一次性 `removeToTrash`。
  - `copyFolderLinks`（`src/lodestar/commands.ts`）：多选文件夹/标签时调 `collectTagsUnder` 收标签，`copyFolderLinks` 用 `pruneCovered` 去重后复制链接。
  - **拖拽多选**（`src/player/tree/index.ts` 的 `TreeDragAndDropController.handleDrag`）：拖选中集合内任意节点，`handleDrop` 把全部移到目标（`moveNode` 消费每个）；三处 LOOSE 拦截被移除（以前防「未分组」拖出根，现在它是真文件夹可随意动）。
- **单测**：`test/lodestar/inbox.test.ts`、`selection.test.ts`、`adapter.test.ts` 覆盖新逻辑。

## 4. 编辑模式 & 注释框（两个 comment controller，别搞混）

- **ambient controller**（id `codeJumpTags`，label 仍叫 "CodeTour"）：在 `player/index.ts` 的 `startPlayer()` 里建。它的 `commentingRangeProvider` 在 `store.isRecording` 时返回整文件范围 → 行号边才有 `+`。
  - 「编辑模式开关」靠在内存里起一个空的 **ambient tour**（`AMBIENT_TOUR_ID`，在 `constants.ts`）来强制建出这个 controller，从而让 `+` 出现；关闭=`endCurrentCodeTour(false)` 拆掉。代码：`toggleEditMode`（`lodestar/commands.ts`）。
- **edit controller**（id `codeJumpTags-edit`）：在 `src/lodestar/editThread.ts` 里**单独**建，专门用来「编辑已有标签」。和编辑模式/录制无关，随时能用。
  - `openTagEditor(tagId, seedBody?)` 在标签所在行开一个 Editing 模式的注释气泡，预填原注释。
  - ✓ 保存=`saveTagEdit`（读 `comment.body` 写回 tag）、✗ 取消=`cancelTagEdit`、🗑 删除=`deleteFromEditor`（进回收站）。
  - 气泡里的按钮靠 `package.json` 的 `comments/comment/context`（✓✗）和 `comments/commentThread/title`（🗑）菜单，`when: commentController == codeJumpTags-edit`。

**「点+在已打标签的行不覆盖、而是基于原注释改」**：`addTag`（`recorder/commands.ts`）里先 `findTagByLocation`，命中就 `openTagEditor(existing.id, 原注释+新输入)`，不再盲写。

**坑备忘**：comment 回复回调里 `thread.dispose()` 必须 `setTimeout(...,0)` 延后，否则提交转圈卡死（见 `addTag`）。

## 5. 删除 / 确认 / 回收站

| 功能 | 代码 |
|---|---|
| 删除前确认 + 「不再询问」 | `confirmDelete()` (`lodestar/commands.ts`)，读/写配置 `codeJumpTags.confirmDelete` |
| 删除 → 进回收站（不真删） | `removeToTrash()` (`tree.ts`)；调用方：`deleteFolder`、`deleteTag` tag 分支、`deleteFromEditor` |
| 回收站列表 + 恢复（**递归展开子文件夹**，**勾选父子级联**，可恢复任意一层节点） | `restoreFromTrash()` (`lodestar/commands.ts`，用 `window.createQuickPick`+`onDidChangeSelection` 实现级联:勾文件夹→子树全勾、子全取消→文件夹取消;`folderRowsDeepestFirst` 自底向上重算父勾选,`updating`+`sameSet` 防回写死循环) + `restoreSelection()`/`removeDescendantById()` (`tree.ts`，把子树里任意节点拔到根) |
| 回收站上限 50、最近在前 | `pushTrash()` (`tree.ts`，`TRASH_LIMIT`) |

**坑备忘**：`QuickPickItem` 自带 `kind` 字段（分隔符用），自定义判别字段别叫 `kind`（我们用 `row`）。
