# 0.7.1 修复「待处理」分组在删除可疑标签后残留 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

> **背景（0.7.0 终审遗留的 Minor #1）：** 用回收站删掉某文件最后一个可疑标签后，运行时可疑注册表（`suspect.ts` 的 `Map`）里仍残留该标签 id，导致树顶「⚠ 待处理」分组继续显示成「待处理 (0)」+「No steps recorded」，直到该文件下一次 recheck（切窗口/编辑/打开）才自愈。根因：删除路径 `batchDeleteByIds`（`src/lodestar/commands.ts`）只做 `removeToTrash + saveStore`，而注册表只有「按文件整体替换」(`setFileSuspects`) 和「整体清空」(`clearSuspects`)，**没有按 id 删除**。本期在删除时把被删标签的 id 从注册表清掉并刷新树，从源头消除残留。

**Goal:** 删除标签/文件夹时，同步把被删标签的可疑注册表条目清掉并刷新树，使「待处理」分组不再残留空壳。

**Architecture:** 给纯模块 `suspect.ts` 增一个按-id 删除的纯函数 `removeSuspects(ids)`；在 `batchDeleteByIds`（删除的单一咽喉，标签与文件夹删除都经它）里，删除前用既有 `collectTagsUnder` 把选区（含文件夹后代）展开成被删标签 id，删除并存盘后调用 `removeSuspects` 清注册表、若有变化则 `refreshTagsTree()`。

**Tech Stack:** TypeScript、vitest、VS Code 扩展 API、webpack。

```ts
// Drop suspect entries for the given ids (e.g. tags just deleted from the
// store). Returns true if any entry was actually removed, so the caller can
// decide whether to refresh the tree.
export function removeSuspects(ids: string[]): boolean {
  let changed = false;
  for (const id of ids) {
    if (registry.delete(id)) changed = true;
  }
  return changed;
}
```

## Task 2: 删除标签/文件夹时清注册表并刷新树

**Files:**
- Modify: `src/lodestar/commands.ts`（`batchDeleteByIds`）

**Interfaces:**
- Consumes: `removeSuspects`（suspect.ts，Task 1）、既有 `collectTagsUnder`（selection.ts，展开选区含文件夹后代为标签节点列表）、既有 `refreshTagsTree`（player/tree，0.7.0 Plan 3 导出）。

**说明：** `batchDeleteByIds` 是删除的单一咽喉——`deleteFolder`、删标签命令都最终走它。`collectTagsUnder(store, ids)` 必须在 `removeToTrash` 循环**之前**调用（节点还在树上才能遍历后代），把含文件夹后代在内的被删标签 id 收集好；删除存盘后再清注册表。`commands.ts` 已有动态 `import("./selection")`（`deleteFolder` 用 `pruneCovered`）与动态 `import("../player/recheck")`（`promoteToOriginal` 用 `recheckFile`）的先例，本任务沿用同样的动态 import 写法。

- [ ] **Step 1: 删除前收集被删标签 id**

`src/lodestar/commands.ts` 的 `batchDeleteByIds`，在确认对话框通过这行之后、`let anyFolder = false;` 这行之前插入收集逻辑。即把：

```ts
  if (!(await confirmDelete(summary))) return false;

  let anyFolder = false;
```

改为：

```ts
  if (!(await confirmDelete(summary))) return false;

  // 删除前（节点还在树上）收集所有被删标签 id，含文件夹后代，
  // 用于删除后从运行时可疑注册表里清掉它们。
  const { collectTagsUnder } = await import("./selection");
  const removedTagIds = collectTagsUnder(store, ids).map(t => t.id);

  let anyFolder = false;
```

- [ ] **Step 2: 删除存盘后清注册表并刷新树**

同一函数末尾，把：

```ts
  if (anyFolder) closeTagEditor();
  await saveStore();
  return true;
}
```

改为：

```ts
  if (anyFolder) closeTagEditor();
  await saveStore();

  // 被删标签若在可疑注册表里有残留，清掉并刷新树，避免「待处理」分组残留空壳。
  const { removeSuspects } = await import("./suspect");
  if (removeSuspects(removedTagIds)) {
    const { refreshTagsTree } = await import("../player/tree");
    refreshTagsTree();
  }
  return true;
}
```
