# Code Jump Tags 0.7.0 设计定稿：original / current 双版本匹配（模型 A）

> 状态：**定稿，待写实现计划**。
> 本稿取代停车稿 `2026-06-24-original-current-matching-design.md`：核心结构沿用模型 A，
> 但 6 个开放问题已全部敲定，且匹配模型经讨论细化为「编辑器内（确定/精确）vs 编辑器外（不确定/模糊）」两套机制。
> 日期：2026-06-29 ｜ 版本目标：0.7.0 ｜ 依赖：0.6.0（`retargetTag` 已就绪）、0.6.1（撤回/恢复已发布）

## 为什么要这套（动机）

标签自动匹配一路在「锚位置 vs 锚内容」之间打补丁，根因是**让一根锚同时承担两个矛盾目标**：
既要随合法编辑刷新（扛重命名），又会因刷新被写烂（污染 → 重开/撤回后跳错行）。

解法：**身份与位置缓存分离**。引入 `original`（对机器只读的身份/裁判）+ `current`（可被写烂、可丢弃的实时内容缓存）。
用户诉求：重命名变量后**点一下**就续命，不用重新打字。

## 核心原则（地基，必须守住）

1. **能被追踪过程改写的东西，没资格当裁判。** 裁判必须是不可被机器改写的那份 = `original`。
2. **`original` 的定义是「对机器只读、只能被人显式确认重设」。** 追踪过程永远写不动它的文字；
   唯一改它的是人的一次显式动作（确认按钮 / 移到光标行 / 建标签）。
3. **三权分立**：身份 `original`（机器只读）/ 位置缓存 `current`（可丢弃）/ 重设权（只在人手里）。
4. **`current` 不当裁判。** 它只有两个用途：① 编辑器内追踪移动的锚 + 编辑器外的第二匹配目标；
   ② 与 `original` 的差异作为「该不该更新 original」的依据。

## 数据模型（`src/lodestar/types.ts`）

`TagNode` 新增 / 沿用：

- **新增 `original?: string`**：建标签 / 人显式确认那一刻的行文字 trim。**对机器只读**，只由 `retargetTag`
  （人显式动作）写。匹配/恢复的唯一裁判。
- **沿用 `text?` 作为 current**（**不改名**，开放问题#2）：这行此刻的真实内容，实时追踪、可被写烂、可丢弃。
- **沿用 `line`**（1-based 实时行号）与 **`pattern`**（URL 深链/兼容，保留不动）。

**回填兼容**（仿 `backfillAnchorText`，在 `persistence.loadStore` 幂等执行一次）：
旧数据无 `original` → 由现有 `text`（或 `patternToText(pattern)`）补上，此后冻结。

## 匹配 / 恢复：两套机制按「确定性」分

整套设计的关键拆分：**你亲眼看见的改动是确定的（精确匹配即可），没看见的改动是不确定的（必须模糊 + 问人）。**

### A. 编辑器内（亲眼看见 → 确定 → 精确匹配，锚 = `current`）

每次文档变更，按标签当前行的遭遇分三种处理。**追踪锚是 `current`（这行此刻的真实内容），不是 `original`。**
理由：物理上被剪切/搬走的是这行现在的内容（current），不是冻住的 original；用 original 去追，一旦 current 已和它分叉就追不到（见「关键边界」）。

1. **就地改了标签那一行（行还在）** → `current` = 这行新内容。`line` 不动。
   → 这是 `current` 与 `original` 开始分叉的地方。
2. **标签那一行被结构性删除（剪切）** → 拿 **`current`** 全文**精确**找（trim 后比较，**距离优先选最近**）：
   - 找到（粘贴点）→ `line` 挪过去；`current` 不变。
   - 没找到（剪了还没粘）→ **先按住** `line`，`current` 不变；下一次编辑/粘贴再扫到时挪过去。
3. **改的是别处** → `line` 按编辑增量平移（`shiftedLine`）。

**编辑器内不跑模糊、不弹可疑态、不打扰。** 因为看得见，精确就够。

### B. 编辑器/窗口外的改动（没看见 → 不确定 → 模糊 + 二重匹配）

触发时机：重开编辑器 / 重开文件 / git pull / 外部工具改 —— 凡是「编辑器没亲眼看见」的改动。
此时行号追踪不可信，走模糊恢复，复用 `resolveLineFuzzy` 的「距离优先同心扩圈」（以实时 `line` 为中心）：

**二重匹配（先裁判后缓存）：**

1. 先拿 **`original`** 模糊找（阈值 0.9）。命中 → 跳/显示到该行，并自愈 `current`。**不打扰。**
2. `original` 找不到 → 再拿 **`current`** 模糊找。命中 → 这就是改名后的行 → 转**软可疑态**，候选 = 该行。
3. 两个都找不到 → 转**硬可疑态**（没候选）。

**用户确认后** → 确认行的文字升格为 `original`（调 `retargetTag`，它同时写 `original`）。

### 阈值（开放问题#1）

**单阈值 `SIMILARITY_THRESHOLD = 0.9`，不新增第二个参数。** 量的是 `original`（或二重匹配里的 `current`）
对**文件真实行内容**的相似度；`current` 字段本身永不进阈值比较，它只是「候选文本」。
就近消歧：模糊匹配走 ring search（中心=实时 `line`），避免跳到全文重复行（如 `#if 0` 宏孪生）。

> 置信带（把「< 0.9 一律问」拆成 silent-heal / ask / lost 三段，需 `T_high`+`T_low`）属 YAGNI，
> 现无手感证据，留待将来从 0.9 加性拆分，不返工。

## 铁律：机器永远不写 `original` 的文字

这条同时是「裁判稳定」和「闭合下面那个关键边界」的根。

- **`original` 的文字只由两件事写：① 建标签时；② 用户显式确认。** 任何机器追踪/恢复都写不动它。
- **编辑器内的「移动」只动 `line`（位置），不动 `original` 的文字。**
  - `current == original` 时移动：内容一样，`original` 在新行天然仍成立。
  - `current != original` 时移动：基于 `current` 追踪，`original` **原地不动**，跟着标签一起被搬到新行，
    留到编辑器外时再让用户拍。
- **编辑器内的就地改内容只动 `current`，不动 `original`。**

### 关键边界（current 与 original 已大不相同时又触发剪切）

场景：`original`=O，就地改成 `current`=C 且 C 与 O 差很大，**用户还没点「C 升 original」**，此时剪切该行再粘贴。

- 若编辑器内追踪用 `original` 去精确找 → 粘贴的是 C、找的是 O → **找不到，标签跟丢**。← 这就是为什么追踪锚必须是 `current`。
- 用 `current`(C) 追踪：剪切→按住→粘贴(C 落在 M)→精确扫到 C 在 M→`line`=M。
  **标签跟到 M，`current`=C，`original`=O 完好**，待确认的分叉随标签搬到新行。✓

## 可疑态长相与动作（开放问题#3，方案 C）

**被动标记（让你知道标签失配，不打扰）：**

- gutter：图标变灰 + `?`（从「锚定」降为「存疑」）。
- 树节点：label 挂 `?` 角标，description 灰字提示，图标变灰。

**hover 卡片（判断材料 + 动作）：**

- 始终两行对照：`original`（原身份）vs `current`（最后追到的）。
- **常驻入口**：只要 `original != current`，hover 常驻 **〔把当前行设为新身份〕** 按钮 —— 这是编辑器内
  「更新 original 的依据」的落点（不弹可疑态，但随手可点）。
- 可疑态时的动作：
  - **〔采纳新位置〕**：`current`（或软可疑的候选行）升 `original`（对应改名 / 更新身份）。
  - **〔移到光标行〕**：手动把标签重指到光标行（经 `retargetTag`，同时写 `original`）。
  - 软可疑：hover 给〔采纳新位置〕（候选行已填好）+〔移到光标行〕；硬可疑：无候选，只给〔移到光标行〕。

> **〔找回原行〕已废弃删除（2026-06-29，实施时发现设计矛盾）。** 原定义：「丢掉 `current`，拿
> `original` 模糊找回真行并自愈」。但它与二重匹配自相矛盾：可疑态的成立前提**就是**「`original` 在
> 文件里找不到」（找得到的话第一步就判健康、自愈、根本不进可疑），而本动作恰恰要「拿 `original` 去
> 文件里找」——两个前提互斥，所以在**任何**可疑标签上它必然扑空。它本想救「`current` 被污染但
> `original` 还在」的场景，可那种场景已被 original-first 自愈、不会变可疑。用户真正要的「在改名/漂移
> 时更新身份」由〔采纳新位置〕(= 把当前行升为新 `original`，与〔把当前行设为新身份〕同一底层操作)
> 覆盖。对应代码 `recoverToOriginal` / `healTagToLine` 已连根删除（commit 见 Plan 2 收尾）。

> 〔把当前行设为新身份〕与〔采纳新位置〕是**同一底层操作**（把 current / 候选行升为 `original`），
> 只是出现语境不同：前者是编辑器内「current 与 original 分叉、未到可疑态」时的常驻入口，
> 后者是编辑器外进入可疑态后的动作。实现上是同一命令，标签文案随语境定。

**树右键**（仅可疑标签出现）：同样给上述动作。

**「待处理」虚拟汇总分组**：树顶一个**只读过滤视图**，汇总全工作区所有可疑标签，便于一次性 triage。
标签仍住其真实文件夹（引用，不搬动）；修完自动移出该分组。

## 触发点（开放问题#4）：可配置 + 只校验当前文件

「编辑器外改动」的重新校验，落成多个独立设置项，每个可单独开关，适配不同人群：

| 设置项（暂名 `codeJumpTags.recheckOn.*`） | 含义 | 底层事件 | 默认 |
|---|---|---|---|
| `focus` | 切出去再切回来（窗口重获焦点） | `window.onDidChangeWindowState(focused)` | 开 |
| `open` | 打开 / 切到某编辑器 | `onDidChangeActiveTextEditor` + `onDidOpenTextDocument` | 开 |
| `externalChange` | 文件被非打字方式改了（git pull / 外部工具） | `FileSystemWatcher` onChange | 开 |
| `save` | 保存时 | `onDidSaveTextDocument` | 关 |
| `idle` | 空闲 N 秒去抖后 | 定时器 | 关 |

**落点尽可能小，两层落实：**

1. **作用域最小**：每个触发只校验「触发它的那个文件」里的标签，不做全工作区扫描。
2. **默认集最小**：默认只开 focus/open/externalChange；save/idle 高级项默认关。

「完全不自动、我自己来」的人可三个全关，靠手动命令（见下）。

## 命令（开放问题#5）

- **新增**：`Code Jump Tags: 重新校验当前文件的标签` —— 重跑 B 的二重匹配/可疑检测，**不受触发点设置影响，永远可用**。
  给「自动触发全关」的人兜底，也供任何人随时手动校验。
- **不新增「随时重设身份」命令**：该需求被 0.6.0 的「移到光标行 / 粘贴到此行」覆盖 ——
  那条 `retargetTag` 在 0.7.0 扩为「同时写 `original`」，把光标放到标签那行执行一次即等于重设身份。

## `retargetTag` 扩展（加性）

现有 `retargetTag(store, id, file, line, text, pattern)` 在 0.7.0 **同时写 `original`**（人显式动作 = 重设身份）。
所有「用户确认」的路径都走它：采纳新位置、移到光标行、撤回/恢复（0.6.1）、找回原行（确认 `original` 不变）。

> **贯穿铁律：用户确认点击的东西，一定能变成 `original`。**

## 与 0.6.x 的衔接 / 净简化

- 0.6.0 的 `retargetTag` 是唯一「人显式重锚」通道，本版扩成同时写 `original`，加性不返工。
- 0.6.1 的撤回/恢复（会话级、针对显式移动）与本版正交，底层同走 `retargetTag`，撤回会顺手还原 `original`。
- **净简化**：现有 `reanchorTag` 里实时跑的那套 `resolveLineFuzzy` + adopt/`tagLineDeleted`/`lineMatchesAnchor`
  防污染体操可大幅删去 —— 编辑器内改为「`current` 精确追踪」，模糊只在编辑器外发生，且比的是不可写的
  `original`/`current`，**结构上没有「污染裁判」这回事**。

## 测试（TDD，纯逻辑先行）

纯函数先写 vitest，再接 VS Code 胶水层：

- **回填**：`original` 缺失时由 `text`/`pattern` 补，幂等。
- **编辑器内追踪（锚=current）**：就地改→更新 current；剪切→按住→粘贴扫到→挪 line；别处改→平移；
  重复内容精确匹配距离优先；空行不锚。
- **关键边界**：current≠original 时剪切粘贴，标签靠 current 跟到新行，original 完好。
- **编辑器外二重匹配**：original 命中→自愈；original 失败 current 命中→软可疑+候选；都失败→硬可疑；
  ring search 就近消歧。
- **铁律**：任何机器路径都不写 original.text；只有确认路径写。
- **可疑集合收集**：「待处理」过滤视图的可疑标签收集（纯函数抽出）。

VS Code 胶水层（触发点接线、hover、右键、命令、待处理分组渲染）薄，逻辑靠上述纯函数覆盖。
现有测试须全绿 + 新增。

## 边界与已知取舍

- **未被亲眼看见、且 current 也没追到的改名**（关窗后 git pull 改了那行，两份都旧）：二重匹配都失配 →
  硬可疑 → 退手动重指（移到光标行）。比「实时重构改名」少见，可接受，死角不漏。
- **重复内容行**（`}`、`i++`）：精确/模糊都靠距离优先选最近；空行/纯符号行无可用锚，保持不锚。
- **置信带**、③ 多链路（同一位置进多个文件夹/各自命名）不在本期；数据模型保持严格树状，为将来留空间。
