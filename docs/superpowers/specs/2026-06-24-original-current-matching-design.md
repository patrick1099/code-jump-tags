## 为什么要这套（动机）

标签自动匹配一路在「锚位置 vs 锚内容」之间打补丁，bug 越修越多，根因是**让两个互相矛盾的
目标共用同一根锚**：那根锚既要随合法编辑刷新（才能扛重命名），又会因刷新而被写烂（污染 →
撤回/重开后跳到错的行）。

讨论中曾考虑两条路：

- **模型 B（单版本 + 标记）**：只存 original（永不可改）+ 实时行号，不要 current 文字副本。
  恢复只认 original；找不到 → 标签变灰 `?`，让你手动 re-base。结构上根本没有能被写烂的东西，
  逻辑极简。代价：重命名标签行后，重开/pull 时它会诚实说"我丢了，帮我重指"，要你**手动重打/重指**。
- **模型 A（双版本，original 当裁判）**：存 original + current。恢复先 original；只有 original 真失效
  （重命名到认不出）才看 current；两者打架就标记可疑、让你定。重命名能**一键**自动续命。代价：
  多一份数据 + 一层"打架判定"。

**用户选 A**，关键诉求：重命名变量后**点一下**就行，不用像 B 那样重新打字。

## 核心原则（这套设计的地基，必须守住）

1. **能被追踪过程改写的东西，没资格当追踪过程的裁判。**
   current 是追踪一路写出来的，它就不可能用来判断追踪自己写没写错。裁判必须是**不可被机器改写**的那份。

2. **original 的精确定义不是"不可变"，而是"对机器只读、只能被人显式点击重设"。**
   追踪过程永远写不动 original；唯一能改它的是人的一次显式动作（= 调 0.6.0 的 `retargetTag`）。
   机器不能贿赂裁判，只有人能重新任命裁判。这一条让「身份稳定」与「能跟重命名走」**同时成立、不矛盾**。

3. **三权分立**：身份 `original`（对机器只读）/ 位置缓存 `current`（可丢弃）/ 重设权（只在人手里）。
   一路那些接缝 bug，在结构上就没地方长了。

4. **current 永远不当裁判、不参与匹配判定。** 它只有两个用途：
   ① 实时显示；② original 哪天"死了"时，那个**已填好、等你一键采纳**的"新身份建议"。

## 数据模型

`TagNode`（`src/lodestar/types.ts`）：

- **新增 `original?: string`**：建标签（或人显式 re-base）那一刻的行文字 trim。**对机器只读**，只由
  `retargetTag`（人显式动作）写。是匹配/恢复的**唯一裁判**。
- **沿用 `text?`** 作为 **current**（实时追踪、可被写烂、可丢弃）。
- **沿用 `line`**（实时行号，增量追踪）与 **`pattern`**（URL 深链/兼容，保留不动）。

> 命名：保留 `text` 字段名当 current（兼容 0.5.x 数据与代码），新加 `original`。若觉得 `text` 名字
> 易混，可在本版改名为 `current`（需同步迁移），列入开放问题。

**回填兼容**：旧数据只有 `text`/`pattern`、无 `original`。在 `persistence.loadStore` 里加一次幂等回填
（仿 0.5.0 的 `backfillAnchorText`）：`original` 缺失时由现有 `text`（或 `patternToText(pattern)`）补上，
此后冻结。

## 匹配 / 恢复策略

**两层分工**（延续 0.5.0 已确立的）：

- **实时编辑（亲眼看见的变动）**：增量行号追踪 `shiftedLine` 更新 `line`，并刷新 `current`(`text`)。
  **不跑模糊**。original 一概不动。
- **冷启动恢复（重开 / git pull / 外部工具改 / 关窗时被改）**：**只用 `original` 去匹配**，复用
  `resolveLineFuzzy` 的「距离优先同心扩圈」（以实时 `line` 为中心，就近取 ≥ 阈值），把 anchor 从
  current 换成 **original**。这样能找回真行，且因 original 不可被污染，**永远不会被写烂的 current 带跑**。

由此得到一个白捡的好处：**current 污染在常见情况下完全无害**——只要那行代码还在，original 自己
就直接命中它了，**压根不会去看 current**。current 的问题只在「original 自己也找不到了」时才浮上来，
而那恰恰是该问人的时刻。

- **original 命中** → 跳/显示到该行，并**自愈** current（把 current 刷成该行当前文字）。
- **original 失配**（行被重命名到认不出 / 行被删）→ 标签转 **可疑态**（见下）。

**就近消歧**：original 匹配仍走 ring search（中心=实时 `line`），避免跳到全文重复行（如 `#if 0` 宏孪生）。

## 可疑态 + 两个动作（A 的灵魂）

**触发时机**：**不**实时盯着 original/current 差异大小（那样你重构改一半它就来烦你），而是
**"重开/pull 时 original 匹配失败"**才浮出可疑态。日常重构期间不打扰，等你回来问一次。

**可疑态长相**（开放，初版建议）：gutter 图标变灰 + 树里该标签加 `?` 角标 + hover 显示两行：
`original（原来是啥）` vs `current（最后追到了啥）`，并给两个动作：

- **「是，采纳 current」**（一键，对应**改名**）：current 升格为新 original（= 拿 current 最后所在行号去调
  `retargetTag`）。以后正常匹配，不再问。
- **「否，以 original 为准」**（对应**误改 / 污染**）：丢掉烂掉的 current，拿 original（它没被污染、还
  描述着真行）去**找回真行并自愈** current。——**这条能成立，正是因为 original 对机器只读。**
  这就是坚持留 original 的全部意义兑现处。

机器**分不清**"改名"与"污染"（数据上长得一模一样），所以这一步**必须人拍**，但无论哪个都只点一下。

## 边界（老实交代）

- **未被亲眼看见的改名**（关窗时 git pull 把那行改了）：current 也没追到，两份都旧 →「是」没东西可
  建议 → 退回**手动重指**（用 0.6.0 的「移到光标行 / 粘贴到此行」，仍是同一个 `retargetTag`）。
  比"实时重构改名"少见得多，可接受。死角不漏。

## 与 0.6.0 的衔接

「采纳 current」「以 original 为准找回」「手动重指」三个动作，**底层都是 0.6.0 的 `retargetTag`**，
只是目标行号来源不同：current 的建议位置 / original 找回的位置 / 你的光标。所以 0.6.0 先把这个原子
操作做扎实，0.7.0 的冲突按钮几乎白捡。`retargetTag` 在本版扩展为**同时写 `original`**（人显式动作→
重设身份），加性、不返工。

## 开放问题（进入实现前要敲定）

1. **相似度阈值**：沿用 0.5.0 的 `SIMILARITY_THRESHOLD = 0.9`？original 匹配是否要更严/更松？
2. **字段命名**：`text` 留作 current，还是改名 `current` 并迁移数据？
3. **可疑态视觉**：gutter 灰 + 树角标 + hover 双行 + 两按钮——具体交互/措辞/按钮落点（hover？树右键？
   一个专门的"待处理"分组？）。
4. **何时重新评估 original 失配**：只在打开文档/可见编辑器变化时？是否对工作区所有标签批量校验，
   还是仅当前文件？（性能 vs 及时性）
5. **是否需要"随时手动 re-base"命令**（不止在可疑态时）：把当前行升级为 original。大概率要，复用 0.6.0。
6. **current 还要不要独立存**：既然不当裁判，是否可只存 original + 实时 line，current 仅运行时算？
   （这其实是滑向 B；保留 current 的唯一理由是"改名时一键建议"。需确认这点价值是否值一份存储字段。）
